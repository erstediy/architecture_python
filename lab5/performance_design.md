# Performance Design — Система управления недвижимостью

## 1. Анализ производительности

### Hot paths
| Endpoint | Причина |
|----------|---------|
| `GET /api/v1/properties?city=...` | Основной поиск, вызывается при каждом открытии каталога |
| `GET /api/v1/properties?min_price=...&max_price=...` | Фильтрация по цене, частый сценарий |
| `GET /api/v1/users/me` | Вызывается при каждом запросе property-service для валидации токена |
| `GET /api/v1/properties/{id}` | Просмотр карточки объекта |

### Потенциально медленные операции
- **Поиск по городу/цене** — полный скан таблицы без кеша
- **Валидация JWT через auth-service** — network round-trip на каждый запрос в property-service
- **Запись на просмотр** — запись в БД + потенциальные конкурентные запросы

### Требования к производительности
- Поиск объектов: p99 < 100ms
- Валидация токена: p99 < 20ms (кешируется)
- Rate limit на поиск: не более 100 запросов/минуту с одного IP

## 2. Стратегия кеширования

### Что кешируем

| Данные | TTL | Стратегия | Инвалидация |
|--------|-----|-----------|-------------|
| Результаты поиска объектов по городу | 60 сек | Cache-Aside | При создании/изменении объекта в этом городе |
| Результаты поиска по цене | 60 сек | Cache-Aside | При создании/изменении объекта |
| Карточка объекта `GET /properties/{id}` | 120 сек | Cache-Aside | При изменении статуса объекта |
| Данные пользователя `/users/me` (в auth-service) | 30 сек | Cache-Aside | Не требуется — только чтение |

### Почему Cache-Aside
Данные меняются редко относительно частоты чтения. Cache-Aside даёт полный контроль над инвалидацией — при обновлении объекта явно удаляем нужные ключи. Read-Through/Write-Through усложнили бы архитектуру без значимых преимуществ.

### Ключи кеша
```
properties:city:{city}           — поиск по городу
properties:price:{min}:{max}     — поиск по цене
properties:id:{id}               — карточка объекта
users:me:{user_id}               — данные пользователя
```

## 3. Стратегия Rate Limiting

### Защищаемые endpoints

| Endpoint | Алгоритм | Лимит | Причина |
|----------|----------|-------|---------|
| `GET /api/v1/properties` | Sliding Window Counter | 100 req/min на IP | Предотвращение массового парсинга |
| `POST /api/v1/auth/token` | Fixed Window Counter | 10 req/min на IP | Защита от брутфорса |

### Выбор алгоритмов

**Sliding Window Counter** для поиска — точнее Fixed Window, сглаживает всплески на границе окна. Реализуется через Redis ZADD/ZRANGEBYSCORE на timestamps.

**Fixed Window Counter** для логина — достаточно для брутфорс-защиты, проще в реализации, строже на коротких интервалах.

### Заголовки ответа
```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 87
X-RateLimit-Reset: 1717200060
```

## 4. Анализ влияния

### Кеширование
- **Cache hit** на поиске по городу снижает нагрузку на БД с O(N) скана до O(1) Redis GET
- **Валидация токена** через кеш: устраняет network round-trip ~5–15ms на каждый запрос property-service
- Ожидаемый cache hit rate при TTL=60s и 100 rps: >90% для популярных городов

### Rate Limiting
- Защищает БД от внезапных всплесков нагрузки
- Клиент получает `429` с `Retry-After` вместо деградации сервиса

### Метрики мониторинга
- `cache_hits_total`, `cache_misses_total` → вычисляемый hit rate
- `rate_limit_rejected_total` — количество отклонённых запросов
- `redis_latency_ms` — задержка самого кеша
- `db_query_duration_ms` — до/после кеширования
